iMocha Security, Compliance & Data Privacy
==========================================

## Security Architecture Overview

iMocha is built with a defense-in-depth security model across all product layers: infrastructure, platform, application, and data. Security is not an afterthought — it is embedded into the Software Development Lifecycle (SDLC) via DevSecOps practices.

### Infrastructure Security

**Cloud Infrastructure (AWS Primary)**
- Production runs on AWS (us-east-1 primary, eu-west-1 Frankfurt, ap-southeast-1 Singapore, ap-south-1 Mumbai)
- Disaster Recovery (GCP us-central1) with RPO < 1 hour, RTO < 4 hours
- All EC2/EKS nodes run in private subnets; no direct internet exposure
- Load balancers: AWS Application Load Balancer with WAF (AWS WAF v2)
  - OWASP Top 10 ruleset enabled
  - Rate limiting: 1,000 requests/5 minutes per IP (configurable per endpoint)
  - Geo-blocking available on request
- DDoS protection: AWS Shield Advanced (managed by iMocha NOC)
- VPC Flow Logs → CloudWatch → Splunk SIEM for real-time alerting

**Network Segmentation**
- Three-tier architecture: Public (ALB, WAF), Application (Kubernetes worker nodes), Data (RDS, ElastiCache — no internet access)
- All cross-tier communication over TLS 1.3 (TLS 1.0/1.1 disabled)
- Security groups follow least-privilege: each service only allows traffic from specific sources on specific ports
- VPC peering to client environments available for Enterprise + On-Premises customers

**Secrets Management**
- All credentials (DB passwords, API keys, TLS certificates) stored in HashiCorp Vault (Vault Enterprise, HA mode)
- No secrets in environment variables, source code, or config files
- Vault policies enforce least-privilege: each microservice has its own policy with access only to its secrets
- Secret rotation automated for: DB credentials (30-day rotation), API tokens (90-day rotation)

### Application Security

**Secure SDLC**
- Threat modeling: all new features reviewed via STRIDE framework during design phase
- SAST: Semgrep runs on every pull request; build fails on HIGH/CRITICAL findings
- Dependency scanning: Snyk scans all Python/Node/Java dependencies daily; critical CVEs patched within 72 hours
- DAST: OWASP ZAP baseline scan on every release candidate
- Container scanning: Trivy scans all Docker images in CI; images with CRITICAL CVEs blocked from deployment
- IaC scanning: Checkov validates all Terraform/Helm before apply

**Authentication & Authorization**
- Multi-factor authentication (MFA): mandatory for all admin users; optional (recommended) for recruiters/hiring managers
- Supported MFA methods: TOTP (Google Authenticator, Authy), WebAuthn/FIDO2, SMS (discouraged)
- Session management: access tokens expire in 15 minutes; refresh tokens in 7 days (rolling); immediate revocation supported
- RBAC: 7 built-in roles (Account Admin, Billing Admin, Assessment Creator, Recruiter, Hiring Manager, Analytics Viewer, Candidate Reviewer); custom roles available in Enterprise
- All privileged admin actions logged in immutable audit trail (CloudTrail + S3 with Object Lock)

**API Security**
- All API endpoints authenticated (no public endpoints except OAuth token endpoint)
- Input validation: Pydantic/Marshmallow schema validation on all request bodies
- SQL injection prevention: ORM-only queries; raw SQL forbidden in application code
- XSS prevention: Content-Security-Policy headers; React DOM escaping; output encoding
- CSRF protection: SameSite=Strict cookies + CSRF tokens for browser-facing endpoints
- API abuse monitoring: ML-based anomaly detection on API call patterns; auto-block on anomalous behavior

## Data Privacy & Residency

### Data Classification

| Class | Description | Examples | Encryption |
|-------|-------------|----------|------------|
| C1 (Public) | Marketing content, docs | Product pages, API docs | None required |
| C2 (Internal) | Business operations data | Internal analytics, staffing data | Encrypted at rest |
| C3 (Confidential) | Client/candidate PII | Names, emails, assessment scores | AES-256 at rest + in transit |
| C4 (Restricted) | Proctoring media, biometric data | Video recordings, face recognition templates | AES-256 + additional access controls + audit logs |

### Data Residency Options

**Default (Multi-tenant SaaS)**:
- Clients select a primary region: US, EU, APAC, or India
- All C3/C4 data remains within the selected region at rest
- Replication to DR region within same jurisdiction (US → US-West, EU → EU-North)

**Dedicated Tenant**:
- Single-tenant databases in client's chosen region
- No shared compute
- Additional regions available: Canada (ca-central-1), Australia (ap-southeast-2), UAE (me-south-1), Brazil (sa-east-1)

**On-Premises**:
- All data stays within client's own network
- No data ever transmitted to iMocha cloud (telemetry can be disabled)
- Requires annual security review agreement with iMocha's security team

### Retention Policies

| Data Type | Default Retention | Minimum | Maximum |
|-----------|------------------|---------|---------|
| Candidate assessment results | 12 months | 3 months | Indefinite |
| Video proctoring footage | 90 days | 30 days | 180 days |
| Audio proctoring recordings | 90 days | 30 days | 180 days |
| Chat/support logs | 24 months | 12 months | 36 months |
| Audit logs | 36 months | 12 months | 7 years (for regulated industries) |
| Deleted candidate data | 30-day soft delete then permanent | N/A | N/A |

### Sub-Processors

iMocha uses the following sub-processors for service delivery:

| Sub-Processor | Purpose | Data Shared | Location |
|---------------|---------|-------------|----------|
| Amazon Web Services | Cloud infrastructure | All data (in-region only) | US/EU/APAC/India |
| Google Cloud Platform | Disaster recovery | Encrypted backups | US |
| Twilio SendGrid | Transactional email | Name, email | US |
| Cloudflare | CDN, DNS, DDoS protection | IP addresses, request metadata (no PII) | Global |
| Stripe | Payment processing | Billing contact info only | US/EU |
| Zendesk | Customer support ticketing | Support contact info | US |

Clients are notified of sub-processor changes with 30 days advance notice (email + in-app notification).

## Incident Response

### Classification
- **P0 (Critical)**: Data breach, platform-wide outage, proctoring system failure
- **P1 (High)**: Partial outage affecting >20% of users, security vulnerability being actively exploited
- **P2 (Medium)**: Degraded performance, non-critical feature outage
- **P3 (Low)**: Minor bugs, feature requests

### Response Times
- P0: Incident Commander engaged within 15 minutes; status page updated within 30 minutes
- P1: Lead engineer engaged within 30 minutes; client notification within 1 hour
- P2: Acknowledged within 4 hours; resolution within 24 hours
- P3: Tracked in roadmap; resolved in next sprint cycle

### Data Breach Notification
- Internal escalation: within 1 hour of confirmed breach
- Client notification: within 24 hours of confirmed breach
- Supervisory authority notification (GDPR): within 72 hours of becoming aware
- Affected individual notification: within 72 hours if high risk

### Security Contact
- Responsible Disclosure / Bug Reports: security@imocha.io (PGP key available)
- Bug Bounty Program: HackerOne private program (invite available to security researchers)
  - Scope: *.imocha.io (excluding demo environments)
  - Payouts: $100–$10,000 depending on severity
  - Hall of Fame: publicly acknowledged researchers

## Proctoring Ethics & Legal Considerations

iMocha's AI proctoring is designed to balance assessment integrity with candidate privacy and dignity.

### What iMocha Proctoring Does
- Face detection (presence verification) — confirms a human is taking the test
- Gaze estimation — detects extended periods of looking off-screen (potential reference material)
- Tab switching detection — browser API-based; no keylogger
- Multiple faces detection — flags if another person enters camera frame
- Audio monitoring (optional) — ambient audio analysis for conversation patterns (no speech-to-text)
- Screen recording (optional) — records screen for post-review; not analyzed by AI in real-time

### What iMocha Proctoring Does NOT Do
- No keystroke logging
- No clipboard monitoring (beyond test-specific copy-paste restrictions)
- No access to candidate's local files
- No microphone-to-text transcription (GDPR/BIPA compliance)
- No biometric identification (face recognition for re-ID only within a session; no cross-session matching or identity database)

### Regional Compliance Notes
- **Illinois BIPA (Biometric Information Privacy Act)**: iMocha collects informed written consent before any proctoring in Illinois. Biometric data deleted after session.
- **EU AI Act**: iMocha's proctoring system is classified as "limited risk" AI; transparency obligations met via candidate disclosure.
- **China Personal Information Protection Law (PIPL)**: Data localization required; iMocha's China region (deployed on Alibaba Cloud in Beijing) stores all data locally.
- **Quebec Law 25**: iMocha publishes a Privacy Impact Assessment for all Quebec-based clients on request.

### Proctoring Dispute Process
1. Candidate or client disputes an AI proctoring flag
2. Human reviewer in iMocha's Integrity Team reviews the relevant footage
3. Decision issued within 2 business days
4. If flag overturned, score remains valid and client is notified
5. Disputed records retained for 6 months post-resolution for appeals
