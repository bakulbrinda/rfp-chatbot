iMocha Security, Compliance & Data Privacy
==========================================
Source documents: Security.docx, Ethical AI.docx, AI Governance.docx

## Security Architecture Overview

iMocha is built with a defense-in-depth security model across infrastructure, platform, application, and data layers. Security controls are embedded into the Software Development Lifecycle (SDLC) via DevSecOps practices.

## Encryption and Data Protection

### Encryption Standards
- Data in transit: TLS 1.2+ (TLS 1.0 and 1.1 are disabled)
- Data at rest: AES-256
- Secrets management: Azure Key Vault (API keys, database credentials, TLS certificates — no secrets in environment variables or source code)

### Secrets Management — Azure Key Vault
iMocha uses Azure Key Vault as its secrets management solution. All sensitive credentials and keys are stored in Azure Key Vault, including:
- API keys and OAuth tokens for third-party integrations
- Database credentials
- TLS/SSL certificates
- Encryption keys

Azure Key Vault ensures that no secrets are stored in environment variables, application source code, or configuration files. This applies to all iMocha environments including integration credentials for UKG, Power BI, and identity provider (SSO) connections.

Q: What does iMocha use for secrets management?
A: iMocha uses Azure Key Vault for secrets management. All API keys, database credentials, and TLS certificates are stored in Azure Key Vault — never in environment variables or source code.

Q: How does iMocha manage its encryption keys and credentials?
A: iMocha manages all encryption keys and credentials through Azure Key Vault. This covers integration API credentials, database passwords, and TLS certificates. No secrets are stored in application code or config files.

### Data Classification
iMocha classifies all data into four tiers:

| Class | Description | Examples | Encryption |
|-------|-------------|----------|------------|
| C1 (Public) | Marketing content, documentation | Product pages, API docs | None required |
| C2 (Internal) | Business operations data | Internal analytics | Encrypted at rest |
| C3 (Confidential) | Client and candidate PII | Names, emails, assessment scores | AES-256 at rest + TLS 1.2+ in transit |
| C4 (Restricted) | Proctoring media, biometric data | Video recordings | AES-256 + additional access controls + audit logs |

## Access Control and Authentication

### Access Control
- Role-based access control (RBAC): administrators, recruiters, hiring managers, and analytics viewers have distinct permission sets
- Principle of least privilege: each user and service has access only to what is required for their function
- Admin access requires multi-factor authentication (MFA)
- All privileged admin actions are logged in an immutable audit trail

### Authentication
- Single Sign-On (SSO): supported via SAML 2.0 and OIDC/OAuth 2.0
- Supported identity providers: Azure AD, Okta (see SSO.docx for full configuration details)
- SCIM 2.0 user provisioning: supported for automated user lifecycle management from Azure AD and Okta

### Session Management
- Access tokens have a short expiry window; refresh tokens support rolling session continuity
- Immediate token revocation is supported for terminated or suspended users

For detailed access control procedures, refer to: Access Control & Termination policy, Acceptable Use policy.

## Network and Infrastructure Security

- All production traffic encrypted with TLS 1.2+
- Web Application Firewall (WAF): OWASP Top 10 ruleset enabled on all production endpoints
- DDoS protection active on all external-facing services
- Network segmentation: public-facing load balancers, application tier, and data tier are separated; no direct internet access to data tier
- Vulnerability management: critical CVEs patched within defined SLA windows; refer to Vulnerability & Patch Management policy
- Penetration testing: annual third-party penetration testing conducted; results reviewed and remediated per defined timelines

For network and infrastructure security details, refer to: Information Security policy, Physical Security policy, Antivirus policy.

## Ethical AI and AI Governance

### AI Governance Framework
iMocha's AI Governance policy governs the design, development, deployment, and monitoring of all AI-powered features including Tara AI (conversational interviews) and Skills Intelligence inference models.

Key governance principles:
- Transparency: candidates are informed when AI is being used in the assessment or interview process
- Human-in-the-Loop: no adverse employment decision is made solely on the basis of an AI output — a human review step is always available
- Bias monitoring: AI models are reviewed for demographic bias during development and monitored post-deployment; bias audits are conducted by iMocha's internal AI ethics team
- Auditability: all AI-generated scores and recommendations are logged with supporting evidence (transcript, score rationale); logs are retained per the Data Retention & Disposal policy
- Explainability: scoring rubrics for AI interview assessments are documented and available to recruiters; candidates can request a review of their AI assessment score

### Ethical AI Principles
iMocha's Ethical AI policy defines the following commitments:
- iMocha AI systems do not make final hiring or employment decisions — they provide recommendations and data to human decision-makers
- Scoring is competency-based; regional accents, communication style variations, and non-native English speakers are not disadvantaged
- Assessment content is reviewed by Subject Matter Expert (SME) panels before deployment to validate relevance, fairness, and freedom from cultural bias
- Few-shot learning examples used in AI model training are reviewed and validated by SMEs for fairness before use

### AI Model Specifications (Tara AI)
- Model: Azure OpenAI GPT-4o
- Methodology: Retrieval-Augmented Generation (RAG), Human-in-the-Loop review, few-shot learning, SME-validated scoring rubrics
- Accuracy: 90% alignment with human interviewer scoring benchmarks
- Interview duration: 5–10 minutes per candidate

For AI governance details, refer to: Ethical AI policy, AI Governance policy, Diversity Equity & Inclusion policy, EEOC Checklist.

## Data Privacy and Compliance

### GDPR
- iMocha is GDPR compliant
- A Data Processing Agreement (DPA) including Standard Contractual Clauses (SCCs) is available for EU clients
- Right to erasure: candidate data deletion requests are processed in compliance with GDPR Article 17
- Data Protection Officer (DPO) is appointed; contact available via iMocha's security team

### Data Retention
- Candidate assessment data: retained for 12 months by default; configurable
- Audit logs: retained for 36 months; extended retention available for regulated industries
- Deleted data: 30-day soft delete followed by permanent deletion

For data privacy details, refer to: Data Classification policy, Data Protection policy, Privacy Policy, GDPR Training, Data Retention & Disposal policy.

### Incident Response
- P0 (Critical — data breach, platform-wide outage): Incident Commander engaged within 15 minutes; status page updated within 30 minutes
- P1 (High — partial outage or active exploit): Lead engineer engaged within 30 minutes; client notification within 1 hour
- GDPR breach notification: supervisory authority notified within 72 hours of confirmed breach

For incident response procedures, refer to: Information Security policy, Business Continuity & Disaster Recovery Plan, Disaster Recovery Testing Report.
