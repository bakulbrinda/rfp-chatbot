iMocha Integrations — Technical Architecture & SSO
===================================================
Source documents: Technical 1.docx, SSO.docx

## Integration Overview

iMocha integrates with Applicant Tracking Systems (ATS), HRIS platforms, identity providers, and business intelligence tools. Integration capability varies by product module (TA vs SI) and by specific system.

IMPORTANT: Only mark a client's system as SUPPORTED if the exact system name appears in this document. Do not assume support for systems not listed here. For unlisted systems, the REST API is available for custom integration.

---

## ATS Integrations (Talent Acquisition)

### Oracle Recruiting Cloud (Oracle ORC)
iMocha has a native integration with Oracle Recruiting Cloud (ORC).

Supported flows:
- Assessment trigger: when a candidate reaches a configured stage in Oracle ORC, iMocha automatically sends an assessment invitation to the candidate
- Score writeback: completed assessment scores (total score, section scores, pass/fail status) are written back to the candidate's application record in Oracle ORC
- Status sync: assessment completion status is reflected in the ORC candidate pipeline

Configuration requirements:
- Oracle ORC tenant admin access required for initial setup
- iMocha implementation engineer coordinates with the Oracle ORC admin during configuration
- Typical setup time: 3–5 business days

---

## HRIS Integrations (Skills Intelligence)

### UKG (UKG Pro)
iMocha integrates with UKG Pro for Skills Intelligence (SI) use cases.

Supported data flows:
- Inbound from UKG: employee roster (name, employee ID, email, job title, department, manager), job changes, terminations, certifications
- Outbound to UKG: validated skill proficiency levels written back to the employee record in UKG Pro (requires UKG custom fields to be configured)

Integration type: bidirectional REST API
Authentication: UKG API credentials stored in Azure Key Vault; not stored in application code or environment variables
Setup: requires UKG system administrator access and iMocha implementation engineer; typical setup time: 3–5 business days

Note: UKG integration is part of the SI product module. It does not apply to TA (pre-hire) workflows unless specifically scoped.

## SSO and Identity Provider Integrations

iMocha supports Single Sign-On (SSO) via SAML 2.0 and OIDC/OAuth 2.0. The following identity providers are supported with documented configuration guides.

### Azure Active Directory (Azure AD)
- Protocol: SAML 2.0 and OIDC
- User provisioning: SCIM 2.0 supported for automated user lifecycle management (create, update, deprovision)
- Role mapping: Azure AD security groups can be mapped to iMocha roles (Admin, Recruiter, Hiring Manager, Analytics Viewer)
- Setup time: 30–60 minutes with IT team following iMocha's Azure AD SSO configuration guide

SCIM with Azure AD:
- SCIM endpoint: provided in iMocha account settings
- Deprovisioned users: deactivated in iMocha (not deleted); assessment history retained per data retention policy
- Group-to-role mapping is configurable in iMocha admin panel

### Okta
- Protocol: SAML 2.0 and OIDC
- User provisioning: SCIM 2.0 supported
- Role mapping: Okta groups mapped to iMocha roles
- Setup time: 30–60 minutes with IT team following iMocha's Okta SSO configuration guide

### SSO Configuration Requirements (Both Providers)
- SP-initiated and IdP-initiated SSO are both supported
- Signed SAML assertions required; assertion encryption is optional but recommended
- iMocha SP metadata (Entity ID, ACS URL, SLO URL) is available in iMocha admin → Settings → SSO
- IdP metadata (Entity ID, SSO URL, certificate) must be provided to iMocha during setup

For detailed SSO configuration steps, refer to: SSO.docx

## Business Intelligence Integration

### Power BI
iMocha supports integration with Microsoft Power BI for custom reporting and visualization.

Supported use cases:
- Skills Intelligence data: aggregate skills inventory, gap analysis data, and upskilling progress published to Power BI via the Power BI dataset API
- TA Analytics data: assessment completion rates, score distributions, and recruiter pipeline metrics available for Power BI consumption

Data refresh: configurable (default: daily)
Authentication: Power BI service account credentials stored in Azure Key Vault

## API Access (Custom Integrations)

For systems not listed above, iMocha provides a REST API for custom integrations.

- API specification: OpenAPI 3.0 (available to clients on request)
- Authentication: OAuth 2.0 client credentials flow (machine-to-machine)
- Webhooks: real-time event delivery for assessment lifecycle events (invitation sent, assessment started, completed, score available)

Common API use cases:
- Custom ATS integration: trigger assessment invitations from any ATS via the API; receive scores via webhook
- Custom HRIS integration: programmatically sync employee data and write skills back to HRIS
- Custom reporting: pull assessment data into a data warehouse for BI analysis

## Technical Architecture Notes

### Encryption in Transit
All API communication uses TLS 1.2+ (TLS 1.0 and 1.1 are disabled on all iMocha endpoints).

### Secrets Management
All integration credentials (OAuth tokens, API keys, SCIM tokens) are stored in Azure Key Vault. Credentials are never stored in environment variables, application code, or config files.

### Data Residency
Data exchanged via integrations remains within the client's configured data region. Cross-region data transfers are not performed without explicit client configuration.
